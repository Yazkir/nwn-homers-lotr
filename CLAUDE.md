# Homer's LOTR VEL v3 — module source

This is an **unpacked Neverwinter Nights 1 (Enhanced Edition) module**, kept in
text form so it can be diffed in git and edited by hand or by an LLM. The
binary `.mod` is built from `unpacked/` on demand.

The module itself ("Legendary LotR" / "Homer's Quest's LOTR") is a Tolkien-themed
persistent-world style adventure built on **CEP 2** (Community Expansion Pack).
It targets NWN ≥ 1.68 and uses a custom TLK named `cep`.

## Workflow at a glance

```
nwn-manager unpack   →  reads NWN modules folder (path stored in
                         .nasher/source), fills unpacked/ from the .mod
                         (overwrites unpacked/ — round-trip is destructive)

edit unpacked/       →  JSON for GFF resources, .nss for scripts
                         commit changes to git as you go

nwn-manager repack   →  builds unpacked/ → dist/<name>.mod, copies into
                         the NWN modules folder
                         (script compilation happens here, errors will surface)

nwn-manager wiki     →  generates a static HTML wiki under docs/
                         (checked in; rebuild with `nwn-manager wiki` when needed)
```

`nwn-manager` lives in the separate [nwn_manager](https://github.com/...) repo
and wraps `nasher` + `nwn_gff` + `nwn_script_comp` (from `niv/neverwinter.nim`).

**Source of truth is `unpacked/` + git.** Don't hand-edit the `.mod`; it'll be
overwritten next repack.

**`docs/` is volatile** — fully regenerated by `nwn-manager wiki`; don't hand-edit it.
- Manual/narrative pages → add/edit in `docs.manual/` (copied into `docs/manual/` on each wiki rebuild). No `<header>`/`<footer>` — the generator injects them. Start `<body>` with `<main>`.
- Generated content changes → modify the wiki generation script in `GIT/nwn_manager`.

**`docs/` is a human-readable reference**: `docs/index.html` (area map), `docs/creatures/`, `docs/areas/`, `docs/conversations/`, `docs/items/`, `docs/stores/`, `docs/factions.html`, `docs/journal.html`. Use it to look up module content before parsing JSON by hand.

### Build artifacts (gitignored, do not commit)

- `dist/` — packed `.mod` output
- `.nasher/` — nasher cache + recorded source path (`.nasher/source`)
- `*.ncs` — compiled scripts; only `.nss` source belongs in git
- `module-index/` — LLM-assist indexes regenerated by `nwn-manager wiki`; **check here before parsing `unpacked/` by hand**

### `module-index/` — precomputed LLM context

All files are regenerated by `bin/refresh-homers-lotr-wiki` (runs `nwn-manager wiki`). They are gitignored and may not exist on a fresh clone — run the wiki once to populate them. They are derived entirely from `unpacked/`, so they reflect whatever the last wiki build saw; re-run after significant edits.

**Use these files instead of reading raw `unpacked/` JSON** when you need resolved names, cross-references, or pre-aggregated views. The raw GFF files in `unpacked/` don't carry resolved TLK strings, and building cross-references (e.g. "where is this item sold?") requires reading dozens of files.

#### Orientation and lookup

| File | Use it when you need to… |
|------|--------------------------|
| `area_index.json` | Find an area by name; see its creature/encounter/store/container counts at a glance; get the list of areas it connects to. Good first stop before opening a `.git.json`. |
| `area_graph.json` | Trace how areas connect (doors, triggers, conversation teleports); check whether two areas are directly linked; understand edge types (door vs. trigger vs. talk). Keyed by area resref with resolved names on every node. |
| `area_paths.json` | Find the fewest-hop route from the Well of Eru (the module's `--path-from` area) to any specific area. Includes a list of areas the path-finder couldn't reach. |
| `creature_index.json` | Look up a creature by name (not resref); get its CR, HP, race, faction, appearance, and every area it appears in. Includes `appearance_id` for cross-referencing with the appearance report. |
| `item_index.json` | Look up an item by name; check its base item, gold cost, and whether players can obtain it; see every acquisition source (sold at / in container / carried by / script). |

#### Conflict and quality reports

These are the files to read when auditing content correctness. They are sorted by severity — most actionable entries first.

| File | Use it when you need to… |
|------|--------------------------|
| `item_tag_conflicts.json` | Find items that share a `Tag` value but differ in properties. Scripts and the engine look up items by tag, so conflicts cause wrong-item-returned bugs. Each entry lists all variants and where they appear. |
| `creature_tag_conflicts.json` | Find creature blueprints whose placed instances diverge (producing synthetic resrefs like `guard__v2`). Signals a blueprint was placed with instance-level overrides — consider whether each variant should be its own `.utc`. |
| `cross_faction_creatures.json` | Find creature blueprints placed in more than one faction. Usually means a faction override was missed on an instance or encounter slot. Small list — quick to review. |
| `inaccessible_items.json` | Find items players can't obtain: carried by NPCs without a droppable flag, or not placed anywhere. Useful for auditing loot completeness or finding dead-end rewards. |
| `unspawned_creatures.json` | Find `.utc` blueprints never placed in any area and never in an encounter pool. Likely orphaned content or work-in-progress. |
| `appearance_faction_report.json` | Audit NPCs that share a visual model (`Appearance_Type`) across factions. The file is sorted with cross-faction appearances first — a creature that looks like a friendly commoner but is in the Hostile faction, or vice versa. Also useful for finding all creatures using a given appearance ID. |

## Source tree (`unpacked/`)

`nasher.cfg` flattens everything into `unpacked/` — no subdirectories.
~7281 files: ~5000 JSON blueprints + ~2340 scripts.

| Pattern              | What it is                                              | Approx count |
|----------------------|---------------------------------------------------------|--------------|
| `module.ifo.json`    | Module config: areas list, event hooks, HAKs, TLK       | 1            |
| `module.jrl.json`    | Journal — quest categories and entries                  | 1            |
| `repute.fac.json`    | Faction list + reputation matrix                        | 1            |
| `<name>.are.json`    | Area static data (tiles, weather, lighting, scripts)    | 287          |
| `<name>.git.json`    | Area "GIT" — placed instances of creatures/doors/etc    | 287          |
| `<name>.gic.json`    | Area instance comments (parallel to .git, comments only)| 287          |
| `<name>.utc.json`    | Creature blueprint                                      | 552          |
| `<name>.uti.json`    | Item blueprint                                          | 2990         |
| `<name>.utd.json`    | Door blueprint                                          | 4            |
| `<name>.utp.json`    | Placeable blueprint                                     | 49           |
| `<name>.utw.json`    | Waypoint blueprint                                      | 49           |
| `<name>.utm.json`    | Merchant/Store blueprint                                | 23           |
| `<name>.utt.json`    | Trigger blueprint                                       | 9            |
| `<name>.ute.json`    | Encounter blueprint                                     | 81           |
| `<name>.dlg.json`    | Conversation/dialogue tree                              | 309          |
| `<type>palcus.itp.json` | Custom toolset palette (creature/door/item/etc)      | 9            |
| `<name>.nss`         | NWScript source                                         | 2340         |

A blueprint's filename **is** its ResRef (e.g. `bree.are.json` → ResRef `bree`).
Areas are 1:1:1 between `.are` (static), `.git` (instances), and `.gic` (per-instance
comments) — keep all three when renaming or deleting an area.

## Quest wiki directives

The wiki's **Quests** section is driven by the **Comment** field of each journal category (set in the NWN Toolset's Journal editor — it is the only per-quest free text; individual entries have no Comment). Add any of the following directives to control how a quest appears on the wiki:

| Directive | Effect |
|-----------|--------|
| `@group 'Name'` | Places this quest under the **Name** heading on the Quests index. Quotes may be single, double, or omitted. Use the same name on all quests in the same group. |
| `@group-order N` | Sets the sort position of this quest's **group** among all group headings (lower = earlier). Set it on any one quest in the group. Groups without it sort alphabetically after numbered groups. |
| `@order N` | Sets the sort position of this quest **within its group** (or globally when no groups exist). Lower = earlier. Quests without `@order` sort alphabetically after numbered ones. |
| `@hidden` | Removes this quest from the wiki entirely — no index row, no detail page. Synonyms: `@retired`, `@inactive`. The journal category stays in the module. |

All directives are case-insensitive. Directive lines are stripped from the human-visible "Builder comment" on quest detail pages.

**Example** — two groups, each with explicit ordering:

```
# Quest Comment for "The Shadow of Mordor" (first quest in Main Story):
@group 'Main Story'
@group-order 1
@order 1

# Quest Comment for "The Mines of Moria" (second quest in Main Story):
@group 'Main Story'
@order 2

# Quest Comment for "The Shire Troubles" (side quest):
@group 'Side Quests'
@group-order 2
@order 1
```

Journal entries within each quest always display in ascending ID order.

After editing Comments in the toolset, run `bin/refresh-homers-lotr-wiki` to rebuild.
